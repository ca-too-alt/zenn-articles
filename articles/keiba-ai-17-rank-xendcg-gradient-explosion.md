## 1. 概要

前回（第16話）では、データリークを排除した環境において、動的特徴量を導入することで穴馬の識別能力（KS統計量 0.50）を確保することに成功しました。しかし、穴馬検知（二値分類）だけでは、レース全体の期待値を最大化するような「適切な順位付け」を行うには不十分です。

第17話では、モデルの目的関数を分類からランキング学習（Learning to Rank: LTR）へと転換し、レース内の全出走馬の相対的な序列を予測する試みについて記述します。その過程で直面した、LightGBMの `rank_xendcg` における「勾配爆発」という数学的課題と、その解決策としてのターゲット設計について解説します。

## 2. 実装内容

馬券戦略、特に期待値に基づく運用を行うには、各馬が1着になる確率だけでなく、レース内での実力の序列を正確に把握する必要があります。そこで、モデルの `objective` を二値分類の `binary` からランキング学習用の `rank_xendcg` へ変更しました。

### 2-1. なぜ rank_xendcg なのか
LightGBMにはランキング学習用として `lambdarank` と `rank_xendcg` の2つの主要な目的関数が用意されています。

- **`lambdarank`**: ターゲットラベルとして「整数（関連度）」を要求します。
- **`rank_xendcg`**: ターゲットラベルとして「連続値（浮動小数点）」を受け入れることが可能です。

本プロジェクトでは、着順そのものだけでなく、走破タイム偏差（パフォーマンススコア）に基づく微細な実力差をラベルに反映させるため、連続値を扱える `rank_xendcg` を採用しました。

### 2-2. ターゲットラベルの設計
各レース（Group）内での相対的な評価を学習させるため、以下の形式でターゲット（正解値）を定義しました。

```python
# model_trainer.py 内でのターゲット生成ロジック（概念）
def create_ranking_target(df, alpha):
    # 着順を反転させ、上位ほど高い値を持つように変換
    # N: 頭数, rank: 実着順
    relevance = (df['num_horses_in_race'] + 1 - df['rank']) * (alpha / 10)
    return relevance
```

## 3. 遭遇した問題

`rank_xendcg` を導入し、初期設定で学習を開始した直後、深刻な数値的不安定性に直面しました。

### 3-1. 天文学的なRMSEと学習の停止
学習ログを確認すると、通常であれば 0.1〜1.0 程度に収まるべき `rmse`（モニタリング用指標）が、`1.85965e+13`（18兆超）という異常な数値を記録していました。同時に、`feature_importance` の値も `1.798997e+32` という浮動小数点の限界に近い値を出力していました。

この状態では、モデルは有効な情報を一切学習できず、スピアマン相関係数は **0.0000**（全くの無相関）となりました。

### 3-2. 勾配爆発の原因特定
原因は、LightGBMのランキング学習における「利得（Gain）」の計算式にありました。
`rank_xendcg` や `lambdarank` は、内部的に各ラベル値 $rel$ に対して以下の計算を行い、利得を算出します。

$$Gain = 2^{rel} - 1$$

もしターゲットラベル $rel$ に、頭数（例：18）などをそのまま、あるいは不適切なスケーリング（$\alpha=20$ など）で渡してしまうと、指数関数の特性により利得が指数爆発します。

例えば、$rel = 30$ の場合、$2^{30} - 1 \approx 1,073,741,823$ となり、勾配（Gradient）が極端に大きくなります。これが連鎖し、モデル内の重みが一瞬で発散（勾配爆発）していたのです。

## 4. 解決アプローチ

この問題を解決するためには、モデルに渡す「リレーション（関連度）」のスケールを、数学的に安定した範囲に収める必要がありました。

### 4-1. ターゲットの正規化
まず、ターゲットラベルを頭数によらず一定の範囲に収めるため、頭数で除算する正規化を導入しました。

```python
# 改修後のターゲット設計
df['target'] = (df['num_horses_in_race'] + 1 - df['rank']) / df['num_horses_in_race']
```
これにより、ターゲットは $0 < target \le 1$ の範囲に限定されます。

### 4-2. モニタリング指標の切り替え
`rank_xendcg` 使用時に `metric='ndcg'` を指定すると、内部的にターゲットを整数にキャストしようとして情報の欠損が発生したり、型エラーを引き起こしたりすることがあります。

これを回避するため、収束のモニタリングには `metric='rmse'` を使用することにしました。ランキング学習において `rmse` 自体の値に大きな意味はありませんが、「値が減少しているか」を確認する指標としては十分に機能します。

## 5. 最終的な解決策

実験 `exp018` において、Optunaを用いて数値的に最も安定し、かつ識別能（スピアマン相関）が最大化される `target_alpha` の範囲を探索しました。

### 5-1. 線形リレーションターゲットの実装
最終的に、以下の線形リレーションターゲットとハイパーパラメータの組み合わせを採用しました。

```python
# model_trainer.py の最終的な実装構成

# 1. target_alpha の探索範囲を 1.1 〜 1.9 に制限（勾配爆発の防止）
target_alpha = trial.suggest_float('target_alpha', 1.1, 1.9)

# 2. ターゲットの生成
# 連続値の情報を保持しつつ、2^rel が爆発しないスケールに調整
df['target'] = (df['num_horses_in_race'] + 1 - df['rank']) * (target_alpha / 10)

# 3. 学習設定
params = {
    'objective': 'rank_xendcg',
    'metric': 'rmse',
    'num_leaves': 127,
    'feature_fraction': 0.7,
    'learning_rate': 0.05,
    'label_gain': [i for i in range(32)] # 明示的に利得テーブルを指定（必要に応じて）
}
```

### 5-2. 導入結果
このターゲット設計への刷新により、数値的不安定性は完全に解消されました。

- **RMSE**: 10兆超 $\rightarrow$ **0.3 〜 0.5**（正常値）
- **スピアマン相関係数**: 0.00 $\rightarrow$ **0.4021**

低迷していた予測性能は劇的に回復し、レース内での全馬の序列を高い精度で予測できる「武器」が完成しました。

## 6. 学んだこと

### 目的関数の数学的背景の理解
「LTR（ランキング学習）は精度が高い」という表面的な情報だけで導入すると、内部的な利得計算（$2^{rel}-1$）のような仕様によって致命的な不具合を招きます。使用するライブラリの目的関数が、内部でターゲットをどのように処理しているかを理解することの重要性を痛感しました。

### ターゲット設計のシビアさ
二値分類（0 or 1）とは異なり、ランキング学習や回帰におけるターゲットの「スケール」は、学習の成否を分ける決定的な要素です。特に指数関数を含む目的関数の場合、正規化やスケーリング（今回における `target_alpha` の制限）は、単なるチューニングではなく「前提条件」となります。

### 定数入力警告への注視
スピアマン相関が 0.0000 になった際の `ConstantInputWarning`（入力が定数であるという警告）は、モデルが勾配爆発によって全サンプルに対して同じ値（あるいは NaN）を出力していることを示す重要なシグナルでした。エラーが出ない「サイレントな失敗」こそ、警告ログに解決のヒントが隠されています。

## 7. 次回予告

ランキング学習の導入により、序列の予測精度は飛躍的に向上しました。しかし、ここでもう一つの課題が浮上します。それは、モデルが更新されるたびに予測値の絶対値が下がる「スコアデフレ」と、それによって固定の買い目閾値が機能しなくなる問題です。

次回、モデルの出力特性の変化を認め、相対的な評価で買い目を決める戦略へと舵を切ります。
